# Bug 记录：开发态 JS 资源版本号漂移导致模块重复请求

## 问题现象
本地运行 `python server.py` 后，首次打开主页时浏览器出现异常多的前端脚本请求：

1. `app.js` 正常加载后，`js/state.js`、`js/dropdown.js`、`js/markdown.js` 等模块被重复请求很多次
2. 同一个模块的请求路径只是在 `?v=` 查询参数上不断变化，例如：
   - `/js/state.js?v=1774183441315416900`
   - `/js/state.js?v=1774183441365931100`
   - `/js/state.js?v=1774183441400262600`
3. 页面虽然最终可能还能渲染出来，但首屏请求数量异常膨胀，调试日志被大量刷屏

## 影响范围
- `routes/static.py`
- 前端模块入口与依赖链：
  - `frontend/index.html`
  - `frontend/app.js`
  - `frontend/js/*.js`

## 已确认的事实
1. 首页会以 ES Module 方式加载 `app.js`
2. 服务端会在返回 JS 文件时，动态把本地 `import './xx.js'` 改写成带 `?v=` 的 URL
3. 开发态原实现中，资源版本号来自 `time.time_ns()`，且是在“每次请求”时重新生成
4. 多个前端模块都依赖 `state.js` 等公共模块，版本号一旦漂移，会被浏览器视为多个不同模块地址

## 复现条件
1. 以开发态启动服务（非桌面 release 模式）
2. 打开主页，加载 `app.js` 和其依赖模块
3. 由于服务端对每个 JS 响应都重新生成一份新的 `?v=`，浏览器在解析模块依赖图时不断发现“新 URL”

## 根因分析
1. **版本号生成粒度错误**
   - `routes/static.py` 中 `_get_frontend_asset_version()` 在开发态直接返回 `time.time_ns()`
   - 这意味着同一次页面加载中，不同 JS 请求拿到的版本号都不一样

2. **服务端改写 import 时放大了问题**
   - `_append_build_id_to_local_js_imports()` 会把模块内的相对 `import` 改写为 `./state.js?v=当前版本号`
   - 当 `config.js`、`chat.js`、`dialog.js` 等文件分别在不同时间点返回时，它们内部指向的 `state.js?v=...` 也全部不同

3. **浏览器按 URL 区分模块实例**
   - 对浏览器来说，`/js/state.js?v=A` 和 `/js/state.js?v=B` 是两个不同模块
   - 结果不仅会重复下载，还可能导致同一逻辑模块被实例化多份，带来状态割裂风险

## 已采用的修复方案
1. **开发态版本号改为“进程级固定值”**
   - 在 `routes/static.py` 模块加载时生成一次 `DEV_FRONTEND_ASSET_VERSION`
   - 开发态后续所有静态资源请求都复用这一固定版本号

2. **保留现有防缓存能力**
   - 每次重启服务，开发态版本号仍会变化
   - 同一次服务进程生命周期内，所有 JS import 的 `?v=` 保持一致

3. **不改动 release 模式构建号逻辑**
   - 桌面 release 模式继续使用 `BUILD_ID`
   - 修复只针对开发态“每请求漂移”的问题

## 关键防回归规则（必须遵守）
1. 对 ES Module 做服务端 `import` 改写时，**版本号必须在一次页面加载或一次服务进程内保持稳定**。
2. 开发态的防缓存策略可以做到“每次启动变化”，但不要做到“每个请求变化”。
3. 若后续继续对 JS/CSS 追加查询参数，必须优先检查：
   - 查询参数是否会在依赖链中层层漂移
   - 同一公共模块是否会被改写成多个不同 URL
   - 浏览器是否会因此重复拉取或重复实例化模块

## 快速验收清单
- [ ] 刷新主页后，`js/state.js` 不再因 `?v=` 不同而被重复请求几十次
- [ ] `app.js` 及其依赖模块在一次页面加载中的版本号前缀保持一致
- [ ] 重启服务后，资源版本号会变化，开发态仍具备防缓存效果
- [ ] 页面初始化、模型下拉、配置面板、聊天发送等前端功能正常
